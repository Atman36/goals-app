## 1. Обзор продукта

### 1.1. Что это

Персональное веб-приложение для постановки, финансирования и достижения целей. Пользователь создаёт цели с изображением, стоимостью (для финансовых) и сроком, пополняет «накопление» по каждой цели одной кнопкой, ведёт журнал и галерею, и видит весь портфель целей на визуальном дашборде.

Ключевое отличие от «копилок» и таск-трекеров: продукт построен на доказательной методологии достижения целей (методичка «Достижение целей: научно обоснованный путь», КПД 1.0) — теория функциональных систем Анохина, WOOP (Эттинген), Implementation Intentions (Голлвитцер), Self-Monitoring (Harkin et al.), формула КПД (Коммуникация → Проявленность → Действие).

### 1.2. Проблема

1. **Intention-behavior gap.** Намерения объясняют лишь ~28% поведения (Sheeran, 2002). Люди ставят цели, но не переходят к действию и не удерживают его.
2. Существующие приложения — либо голые копилки (только сумма), либо таск-менеджеры (только задачи). Ни одно не соединяет: **визуальный образ будущего + финансовый прогресс + программу действий + обратную связь**.
3. Цели «в голове» не имеют формы: нет образа, нет цифр, нет свидетелей — нет реализации.

### 1.3. Решение (маппинг на 4 звена функциональной системы)

| Звено (Анохин) | Техника из методички | Реализация в продукте |
|---|---|---|
| 1. Афферентный синтез — формирование цели | Self-concordance check, Goal-Setting Theory (конкретность, срок, измеримость) | Визард создания цели: обязательные срок/название (+ сумма для финансовых), опциональный чек самосогласованности (4 вопроса) |
| 2. Акцептор результата — образ будущего | WOOP / Mental Contrasting | Обязательное изображение цели, галерея-визуализация, WOOP-блок в карточке цели |
| 3. Программа действия | Implementation Intentions (if-then) | Чек-лист «всё необходимое для достижения» + if-then планы с триггерами |
| 4. Обратная афферентация — сличение | Self-Monitoring, Habit Anchoring | Кнопка быстрого пополнения / отметка пунктов чек-листа, прогресс в %, история операций, требуемый темп, недельная рефлексия |
| Усиление | Формула КПД, публичность | Шаринг цели / партнёр по подотчётности (фаза 3) |

### 1.4. Целевая аудитория

**Продукт персональный** — единственный пользователь (владелец), продукт строится под собственные сценарии. Мультипользовательская модель, тарифы, командные функции — вне скоупа. Архитектурно данные всё равно изолируются по `user_id` (RLS): это почти бесплатно и оставляет дверь открытой, если продукт когда-либо станет публичным.

### 1.5. Зафиксированные продуктовые решения

| Вопрос | Решение |
|---|---|
| Для кого продукт | Персональный, для себя. Никаких ролей, тарифов, админки для «клиентов» |
| Валюты | Две: **₽ (RUB)** и **$ (USD)**. Валюта задаётся на уровне цели |
| Нефинансовые цели | **Включены в MVP**: без стоимости, прогресс по чек-листу + срок |
| Автоимпорт пополнений (банки/СБП) | Дальний roadmap (Фаза 4+), в архитектуре не блокируем |
| Telegram Mini App как второй клиент | Рассматривается после веб-MVP (Фаза 4); веб строим API-ready |

## 2. Пользовательские сценарии (JTBD)

1. «Когда я решаю копить на что-то, я хочу превратить желание в оформленную цель с картинкой, суммой и сроком, чтобы она стала реальной, а не оставалась мыслью».
2. «Когда у меня появились свободные деньги, я хочу за 5 секунд добавить сумму к цели, чтобы не терять контакт с прогрессом».
3. «Когда я открываю приложение, я хочу за один взгляд понять состояние всех целей: сколько накоплено, сколько осталось, успеваю ли по срокам».
4. «Когда моя цель не про деньги (запустить проект, пройти обучение, оформить документы), я хочу вести её в той же системе: образ, срок, чек-лист шагов и прогресс».
5. «Когда цель "буксует", я хочу понять, чего не хватает — денег, действий или живого образа, — чтобы скорректировать план, а не бросить».
6. «Когда мне нужна мотивация, я хочу открыть галерею образов всех моих целей и прожить картину будущего».
7. «Когда я достигаю цели, я хочу зафиксировать это и увидеть историю пути (взносы, отметки, комментарии, фото)».

## 3. Функциональные требования

### 3.0. Типы целей — P0

| | Финансовая цель | Нефинансовая цель |
|---|---|---|
| Обязательные поля | Название, изображение, стоимость, валюта (₽/$), срок | Название, изображение, срок |
| Прогресс | `накоплено / стоимость` | `выполнено пунктов чек-листа / всего пунктов` |
| Основное действие | Кнопка «+ Добавить сумму» | Отметка пункта чек-листа |
| Чек-лист | Дополнение к деньгам («готовность») | Основной механизм прогресса |
| История | Взносы/списания | Отметки пунктов, комментарии |
| Индикатор темпа | «Откладывайте X/мес» | «Осталось N пунктов, ~M пунктов/нед» (P1) |

Правила для нефинансовых целей:
- Пока чек-лист пуст — прогресс 0%, карточка показывает обратный отсчёт до срока и призыв «Добавьте шаги».
- Ручная корректировка прогресса слайдером (для целей, где чек-лист не отражает объём) — P1.
- Тип цели задаётся при создании; конвертация типа после создания — P2 (в MVP не нужна).

### 3.1. Дашборд целей (главный экран) — P0

Сетка карточек всех активных целей (оба типа вперемешку, визуально различимы бейджем/иконкой). Требования к карточке:

- **Изображение цели** — hero-элемент карточки (cover, с плавным hover-эффектом).
- **Название цели.**
- **Прогресс-бар / кольцо прогресса** с процентом (по правилам типа цели, см. 3.0).
- **Финансовая:** «125 000 ₽ из 300 000 ₽» или «$1 250 из $3 000» — в валюте цели. **Нефинансовая:** «5 из 8 шагов».
- **Дата (дедлайн)** и индикатор темпа: осталось N мес.; статус — «в графике» / «нужно ускориться» (см. 3.3.4).
- **Быстрое действие на карточке:** финансовая — `[+ Добавить сумму]`; нефинансовая — переход к чек-листу.
- Клик по карточке → страница цели.

Дополнительно на дашборде:

- **Сводка сверху (агрегаты):** всего целей; финансовые агрегаты — **раздельно по валютам**, двумя строками: «₽: накоплено 340 000 из 900 000» и «$: накоплено 1 200 из 5 000». Кросс-курсовая свёртка в общий эквивалент — P2 (см. 3.9). Нефинансовые — «выполнено X из Y шагов по всем целям».
- **Сортировка:** по дедлайну (default), по проценту, по дате создания, ручная (drag-n-drop — P2).
- **Фильтры:** активные / достигнутые / архив; по типу (финансовые / нефинансовые); по валюте.
- **Кнопка «+ Новая цель».**
- Мягкая рекомендация из методички: при создании 6-й+ активной цели — ненавязчивое предупреждение «3–5 активных целей — оптимум» (не блокирует).
- Пустое состояние: онбординг с предложением создать первую цель.

### 3.2. Создание / редактирование цели — P0

Пошаговый визард (необязательные шаги можно пропускать):

**Шаг 0. Тип цели:** финансовая / нефинансовая (две крупные карточки-выбора).

**Шаг 1. Основа (обязательный):**
- Название (конкретное, 3–60 симв.).
- Изображение: загрузка файла (drag-n-drop, до 10 МБ, jpg/png/webp) — обязательно. Пока не загружено — плейсхолдер-градиент.
- Для финансовой: стоимость цели (целое число > 0) + **валюта: ₽ или $** (переключатель; default — из настроек профиля). Уже накоплено на старте (default 0).
- Дедлайн (дата).
- Описание (опционально, markdown-lite).

**Шаг 2. Проверка «своя ли цель» (опциональный, рекомендуемый):**
- 4 вопроса self-concordance (интерес / ценности / стыд-вина / внешнее давление), каждый — шкала 1–5.
- Если доминируют вопросы 3–4 — мягкое предупреждение с предложением переформулировать (текст из методички).
- Результат сохраняется в карточке цели.

**Шаг 3. WOOP (опциональный):**
- 4 поля: Wish (3–6 слов), Outcome (лучший исход), Obstacle (внутреннее препятствие; подсказка: «не "нет времени", а что внутри вас»), Plan (если [триггер] → то [действие]).
- Сохраняется как WOOP-запись цели.

**Шаг 4. Чек-лист «всё необходимое» (для нефинансовых — настойчиво рекомендуемый):**
- Список пунктов помимо денег (документы, действия, покупки, договорённости). Для нефинансовой цели визард предлагает добавить минимум 3 пункта сразу.

После создания цели валюта не меняется, если по цели уже есть взносы (иначе ломается арифметика); без взносов — менять можно.

### 3.3. Страница цели (детальная) — P0

Полноэкранная страница с блоками:

**Хедер:**
- Крупное изображение цели (hero).
- Название, описание, дедлайн, тип, статус (активна / достигнута / архив).
- Крупный прогресс: кольцо/бар, %; для финансовой — «накоплено из стоимости», «осталось X»; для нефинансовой — «выполнено X из Y шагов».
- Действия: редактировать, отметить достигнутой, архивировать, удалить (с подтверждением; удаление — soft delete, см. раздел 8).

**3.3.1. Быстрое пополнение (только финансовые) — P0 (ключевая механика)**
- Кнопка `[+ Добавить сумму]` открывает попап/шит:
  - поле суммы в валюте цели (numpad-friendly на мобильном);
  - пресеты: для ₽ — +500, +1 000, +5 000, +10 000; для $ — +10, +50, +100, +500 (настраиваемые — P2);
  - опциональный комментарий к взносу («премия», «кэшбек»);
  - дата взноса (default — сегодня, можно задним числом);
  - отрицательные корректировки (снял/потратил) — отдельный тип «списание».
- После добавления: **оптимистичное обновление UI** — прогресс-бар анимированно растёт, конфетти при пересечении порогов (25/50/75/100%).
- Мутация идемпотентна (client-generated ID операции): повторный сабмит при плохой сети не задваивает взнос.
- Все взносы пишутся в журнал операций.

**3.3.2. История операций — P0**
- Финансовая цель: лента взносов/списаний — дата, сумма (+/–), комментарий. Редактирование/удаление операции (soft delete, пересчитывает итог).
- Нефинансовая цель: лента событий — «пункт выполнен/снят», комментарии.
- Мини-график накопления/прогресса по времени (line/area chart) — P1.

**3.3.3. Чек-лист «всё необходимое для достижения» — P0**
- Пункты с чекбоксами: название, опциональные заметка и срок.
- Типы пунктов (теги): «действие», «документ», «покупка», «договорённость», «if-then план».
- Для типа «if-then план» — структурированная форма: триггер («если…») + действие («то я…») + тип плана (инициация / сохранение курса / защита от срыва).
- Финансовая цель: прогресс чек-листа отображается отдельно от финансового — «Готовность: 4/7 пунктов». Нефинансовая: чек-лист = основной прогресс.
- Комбинированный индикатор для финансовых (финансы + готовность) — P1, опционально в настройках.

**3.3.4. Индикатор темпа — P1**
- Финансовая: `требуемый темп = (стоимость − накоплено) / месяцев_до_дедлайна` → «Чтобы успеть к 01.06.2027, откладывайте ~14 500 ₽/мес». Сравнение с фактическим средним темпом за 3 мес. → «в графике / отстаёте / опережаете».
- Нефинансовая: `осталось пунктов / недель до дедлайна` → «~2 пункта в неделю».

**3.3.5. Комментарии / журнал цели — P0**
- Лента текстовых заметок с датой (markdown-lite, эмодзи).
- Возможность прикрепить фото к комментарию (попадает и в галерею цели).
- Использование: рефлексия, фиксация срывов и инсайтов; «протокол после срыва» из методички — как подсказка-шаблон (P2).

**3.3.6. Галерея цели — P0**
- Сетка изображений (masonry/grid), загрузка нескольких файлов, подписи.
- Любое изображение можно назначить обложкой цели.
- Лайтбокс-просмотр со свайпом.
- Назначение: усиление акцептора — визуализация образа результата.

**3.3.7. WOOP-блок — P1**
- Отображение сохранённого WOOP (W/O/O/P), редактирование, дата последнего проживания.
- Напоминание «вернуться к образу» при отсутствии активности по цели > 14 дней (P2).

### 3.4. Общая галерея — P0

Отдельная страница `/gallery`:
- Все изображения всех целей единой masonry-сеткой («карта визуализации» / vision board).
- Фильтр по цели, подписи, лайтбокс.
- Режим «слайдшоу» на весь экран для проживания образов — P2.

### 3.5. Достижение цели — P1

- Кнопка «Цель достигнута» → экран празднования (анимация, итоги: сколько копил / сколько шагов прошёл, сколько операций, график пути).
- Достигнутая цель уходит в раздел «Достигнутые» — библиотека успехов, усиливающая память системы (звено 1).

### 3.6. Рефлексия — P2

- Еженедельная рефлексия (шаблон из Приложения D методички): что обещал / что сделал / что мешало / один if-then план на следующую неделю.
- Напоминание раз в неделю (день/время настраиваются).
- История рефлексий на отдельной странице.

### 3.7. Подотчётность и шаринг (формула КПД) — P2

- Публичная read-only ссылка на цель (прогресс + образ, без сумм — настраивается).
- «Партнёр по подотчётности»: приглашение по ссылке; партнёр видит прогресс и еженедельные отчёты.
- Для персонального продукта достаточно read-only ссылки; полноценный аккаунт партнёра не делаем, пока не понадобится.

### 3.8. Аутентификация и профиль — P0

- Вход: email + magic link, OAuth (Google). Продукт персональный — регистрация фактически закрыта (allowlist из одного email в конфиге).
- Профиль: имя, валюта по умолчанию для новых целей (₽/$), тема (light/dark), день недельной рефлексии.

### 3.9. Мультивалютность — P0 (₽ и $)

- Валюта — атрибут **цели**, не пользователя. Все суммы цели (стоимость, взносы, история, темп) — строго в её валюте.
- Форматирование: `Intl.NumberFormat('ru-RU', {currency})` — «125 000 ₽», «$1,250».
- Агрегаты на дашборде — раздельно по валютам (MVP). Никаких конвертаций внутри цели.
- **P2: общий эквивалент портфеля** в базовой валюте профиля. Источник курса: ЦБ РФ (публичный API) с кешированием на сутки, fallback — ручной курс в настройках. Эквивалент — справочный, помечается «≈».
- Архитектурно список валют — enum, расширяемый без миграции логики (третья валюта — вопрос конфигурации).

## 4. Модель данных

```
User
  id, email, name, avatar_url
  default_currency: enum('RUB','USD')
  theme, reflection_day, created_at

Goal
  id, user_id → User
  kind: enum('financial','non_financial')
  title, description
  cover_image_id → MediaItem (nullable)
  currency: enum('RUB','USD') (nullable для non_financial)
  target_amount (bigint, минорные единицы; nullable для non_financial)
  initial_amount (bigint, default 0; nullable для non_financial)
  manual_progress (smallint 0–100, nullable; только non_financial, P1)
  deadline (date)
  status: enum('active','achieved','archived')
  achieved_at (nullable)
  self_concordance (jsonb, nullable)      -- ответы на 4 вопроса
  sort_order (int)
  created_at, updated_at, deleted_at (nullable, soft delete)

Contribution                              -- взносы/списания (только financial)
  id (uuid, client-generated — идемпотентность)
  goal_id → Goal
  amount (bigint, может быть отрицательным)
  note (nullable), occurred_at (date)
  created_at, deleted_at (nullable, soft delete)

ChecklistItem
  id, goal_id → Goal
  title, note (nullable), due_date (nullable)
  kind: enum('action','document','purchase','agreement','if_then')
  if_then (jsonb, nullable)               -- {trigger, action, plan_type}
  is_done (bool), done_at (nullable)
  sort_order, created_at, deleted_at (nullable)

Comment
  id, goal_id → Goal
  body (text), created_at, updated_at, deleted_at (nullable)

MediaItem
  id, goal_id → Goal (nullable)
  comment_id → Comment (nullable)
  storage_path, width, height, blurhash
  caption (nullable), created_at, deleted_at (nullable)

WoopEntry
  id, goal_id → Goal
  wish, outcome, obstacle, plan
  last_lived_at (nullable), created_at, updated_at

Reflection                                -- P2
  id, user_id, week_start (date)
  promised, done, blocked, new_if_then
  created_at

FxRate                                    -- P2 (для эквивалента портфеля)
  id, base ('USD'), quote ('RUB'), rate (numeric), source ('cbr'|'manual'), fetched_at
```

**Вычисляемые значения (не хранить, считать):**
- Финансовая: `saved = initial_amount + SUM(contributions.amount where deleted_at is null)`; `progress = saved / target_amount`.
- Нефинансовая: `progress = done_items / total_items` (или `manual_progress`, если задан — P1).
- `required_pace = (target_amount − saved) / months_to_deadline`.

**Инварианты (проверяются на backend, Zod + constraints):**
- `financial` ⇒ `currency`, `target_amount > 0` обязательны; `non_financial` ⇒ `currency`, `target_amount`, `initial_amount` = NULL, Contribution запрещены.
- Все денежные суммы — bigint в минорных единицах (копейки/центы), никаких float.
- Смена валюты цели запрещена при наличии хотя бы одной не удалённой Contribution.

### 5.2. Хостинг — два сценария

1. **Vercel + Supabase Cloud** — нулевой DevOps, preview-деплои. Риск: оплата/доступность из РФ.
2. **VPS (Timeweb Cloud / Selectel) + Docker + Coolify, self-hosted Supabase или Postgres + S3-совместимое хранилище (Selectel S3 / VK Cloud)** — полный контроль, дешевле на дистанции.

Рекомендация: MVP на Vercel + Supabase (скорость), архитектурно не завязываться на vendor-специфику (Drizzle + стандартный S3-интерфейс через абстракцию), чтобы миграция на VPS была днём работы.

### 5.3. Готовность к Telegram Mini App (Фаза 4)

Чтобы TMA потом не потребовал переписывания:
- Вся бизнес-логика — в слое `lib/actions` + `lib/db/queries`, не в компонентах. Server Actions — тонкие обёртки.
- Параллельно Server Actions заводим **route handlers `/api/v1/*`** для ключевых операций (goals, contributions, checklist) — это же пригодится для будущих интеграций (банки/СБП).
- Auth-слой абстрагирован: сессия Supabase сегодня, Telegram initData-валидация завтра — через единый `getCurrentUser()`.
- Дизайн-токены и компоненты прогресса/карточек — переносимы (без завязки на layout десктопа).

## 6. Требования к UI/UX

- **Визуальная планка — высокая.** Дашборд должен вызывать желание его открывать: изображение — главный элемент, крупная типографика, воздух. Продукт персональный → дизайн-система своя (или форк наработок money.shamenkov.com как стартовая точка — переиспользование собственного актива без внешних обязательств).
- **Mobile-first.** Основной сценарий (quick-add, отметка пункта) чаще происходит с телефона. Bottom-sheet, крупные тач-таргеты, numpad для сумм.
- Тёмная и светлая темы.
- Финансовые и нефинансовые цели визуально едины, различаются бейджем и содержимым нижней части карточки — не два разных продукта.
- Анимации осмысленные: рост прогресса, празднование порогов, shared-layout переходы.
- Скелетоны и blurhash вместо спиннеров.
- Локаль: русский; форматирование — `Intl.NumberFormat('ru-RU')` для обеих валют.
- Доступность: контраст AA, фокус-стейты, prefers-reduced-motion.
- Пустые состояния прописаны для всех экранов (дашборд без целей, цель без чек-листа, галерея без фото).
- Ошибки объясняются человеческим языком; необратимых действий без подтверждения нет.

## 7. Нефункциональные требования

- **Производительность:** LCP дашборда < 2.0 с на 4G; quick-add и чекбоксы — отклик UI < 100 мс (оптимистично).
- **Надёжность денег:** суммы — bigint в минорных единицах; операции идемпотентны (client-generated UUID); удаление денежных операций — только soft delete.
- **Приватность:** RLS на всех таблицах (изоляция по user_id, даже при одном пользователе); публичный шаринг — только явный opt-in (P2); чувствительные данные не пишутся в логи.
- **Резервные копии БД:** ежедневные автоматические + проверка восстановления раз в квартал.
- **Лимиты медиа:** до 10 МБ/файл, jpg/png/webp по сигнатуре (не по расширению), до 50 изображений на цель (MVP); server-side ресайз до 2560px.
- **Uptime-ориентир:** 99.5% (персональный продукт, без формального SLA); error rate < 1%.

## 8. Инженерная готовность и эксплуатация

Раздел адаптирует стандартный «архитекторский» набор (RBAC, audit, observability, admin, support-mode и т.д.) под реальность персонального продукта: берём то, что защищает данные и деньги и даёт обратную связь, — сознательно выбрасываем enterprise-обвязку.

### 8.1. Что берём

| Компонент | Реализация в этом продукте | Приоритет |
|---|---|---|
| **Observability** | Sentry (ошибки front/server), structured logging (pino) в Server Actions и route handlers, correlation по requestId; алерт на всплеск ошибок (email/Telegram) | P0 |
| **Журнал критичных операций (audit lite)** | История денежных операций и отметок — уже часть домена (Contribution, done_at). Плюс `deleted_at` везде и `updated_at` на Goal. Отдельная audit-таблица «кто/что/до/после» не нужна — пользователь один | P0 |
| **Идемпотентность** | Client-generated UUID для Contribution; повторный сабмит формы не задваивает деньги | P0 |
| **Soft delete + retention** | `deleted_at` на Goal, Contribution, ChecklistItem, Comment, MediaItem; «удалённое» скрыто, физическая очистка — вручную/крон через 90 дней | P0 |
| **Валидация на backend** | Zod-схемы в каждом Server Action / route handler; инварианты из раздела 4 — constraints в БД | P0 |
| **Бэкапы и восстановление** | Ежедневный бэкап Postgres; задокументированная процедура restore; медиа — версионирование бакета | P0 |
| **Продуктовая аналитика (self-analytics)** | PostHog: события из 8.4. Смысл для персонального продукта — видеть собственное поведение с целями (частота взносов, брошенные цели) и проверять гипотезы методички | P1 |
| **Feature flags (лёгкие)** | Не сервис, а конфиг: env/таблица `app_config`. Достаточно, чтобы выкатывать сырые фичи (рефлексия, FX-эквивалент) выключенными | P1 |
| **Фоновые задачи** | Vercel Cron / pg_cron: очистка soft-deleted, обновление FX-курса (P2), напоминание о рефлексии (P2) | P1–P2 |
| **API-версионирование** | Только для `/api/v1/*` (задел под TMA/интеграции); внутри веба Server Actions не версионируем | P1 |

### 8.2. Что сознательно НЕ берём (и почему)

| Компонент | Почему не нужен |
|---|---|
| RBAC / permission matrix / UI управления ролями | Пользователь один. «Права» = ownership-проверка через RLS. Точка |
| Админ-панель | Supabase Studio + SQL покрывают 100% потребностей владельца-разработчика |
| Support-mode, impersonation, теги обращений | Поддержка = сам себе; Sentry-issue вместо тикета |
| SLA-мониторинг бизнес-процессов, очереди с дашбордами | Нет процессов с SLA; фоновых задач — единицы |
| Webhooks наружу, API keys, публичный API | До интеграций (Фаза 4+) — мёртвый код. Задел — `/api/v1` |
| NPS/CSAT, churn-аналитика, тарифные события | Нет клиентов и тарифов |
| Массовые операции, импорт | Объём данных — десятки целей; экспорт (см. ниже) достаточно |

Единственное из «выброшенного», что добавляем в облегчённом виде: **экспорт данных** (JSON + CSV взносов по кнопке в настройках, P1) — это страховка владения данными, а не enterprise-фича.

### 8.3. Definition of Done для фичи (адаптированный)

Фича готова, если:
- Сценарий реализован полностью, включая пустые состояния и ошибки.
- Валидация и ownership-проверка — на backend (не только в UI).
- Денежные/разрушающие операции: идемпотентность, soft delete, подтверждение в UI.
- Есть миграция БД и (для money-логики) юнит-тесты расчётов.
- Ошибки попадают в Sentry с контекстом; чувствительное — нет.
- Добавлены аналитические события (если фича попадает в 8.4).
- Сырая фича закрыта флагом из `app_config`.
- Rollback: миграции обратимы, фича отключаема флагом.

- Статусы: активна / достигнута / архив.

### Фаза 4 — Второй клиент и интеграции (дальний roadmap)

## 10. Метрики успеха

Продукт персональный, поэтому метрики — это **self-analytics** и проверка гипотез методички, а не growth-воронка.

- **Ядро (обратная афферентация):** медианная частота операций (взнос или отметка пункта) на активную цель — целевое значение ≥ 2/мес; % целей с активностью за последние 14 дней.
- **Использование:** количество открытий дашборда в неделю; доля недель с ≥1 операцией.
- **Достижение (north star):** % целей, доведённых до 100% (по типам отдельно); медианное время жизни достигнутой цели.
- **Методология (v1+):** доля целей с WOOP/if-then; сравнение частоты операций у целей с WOOP и без (личная проверка гипотезы Эттинген/Голлвитцера).
- **Технические:** error rate < 1%, crash-free ≈ 100%, P95 Server Action < 500 мс.

## 11. Открытые вопросы

1. **FX-эквивалент (P2):** достаточно ли справочной строки «≈ всего в ₽» на дашборде, или нужен переключатель базовой валюты ₽/$?
2. **Комбинированный прогресс финансовой цели** (деньги + готовность чек-листа): показывать двумя независимыми индикаторами (проще и честнее) или одним взвешенным (нужна формула весов)?
3. **Ручной прогресс нефинансовых целей:** точно P1, или есть цели, где он понадобится с первого дня (тогда поднять в MVP)?
4. **Хостинг на старте:** Vercel + Supabase Cloud или сразу VPS? Влияет на первую неделю работ.
5. **Нейминг и домен:** зафиксировать рабочее название до старта UI (влияет на логотип, PWA-манифест, тон текстов).
